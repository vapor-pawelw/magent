# Sidebar Row Stability

## User-Facing Behavior

- Selecting a thread should not change the sidebar width.
- Selecting a thread should not make a multiline thread row grow or shrink.
- Task descriptions should keep the same wrapping/height while selection state changes.
- Thread row heights are dynamic based on content (description lines, subtitle, PR row) with a per-thread minimum enforced by `ThreadCell.sidebarRowHeight`. Height must not change when selection state toggles.
- Pin, rate-limit, keep-alive, and duration badges sit on the capsule border and should not cause row-width or height changes on state transitions.
- Periodic sidebar refreshes should not scroll the thread list back to the top while the user is reading older rows.
- Periodic sidebar refreshes should not yank the current selection back into view if the user has intentionally scrolled elsewhere in the list.
- On app launch, once the sidebar has finished loading and layout has settled, the selected thread may be centered once so startup focus lands on the active context.

## Implementation Notes

- Keep the main split view structure stable: swap detail content inside a persistent container instead of removing and re-adding split-view items.
- Preserve the sidebar width during detail-content swaps with `SplitViewController.preserveSidebarWidthDuringContentChange(...)`.
- Load the saved sidebar width into `preferredSidebarWidth` before startup selection/detail work begins, and apply the initial divider position no later than `viewWillAppear`. If launch-time content swaps run first, AppKit will build sidebar rows against the default width and then reflow them again when the saved width is restored.
- Route thread-row height through `ThreadCell.sidebarRowHeight(descriptionLines:hasSubtitle:hasPRRow:narrowThreads:)` which computes per-thread height from actual content with a minimum enforced by `minimumContentHeight(narrowThreads:)`.
- Dynamic heights are computed in `heightOfRowByItem` using `ThreadCell.estimatedDescriptionLineCount` for text-width-based line estimation.
- Keep the description line limit in `AppSettings.sidebarDescriptionLineLimit` so the `Narrow threads` toggle switches both text wrapping and minimum row height together.
- Keep description text style stable for description rows (semibold) so wrapping does not change with unread-selection state transitions.
- Status badges (pinned, rate-limit, keep-alive, duration) are now positioned on the capsule border rather than in a trailing stack. They do not affect available text width.
- Refit sidebar outline width from the scroll view's visible clip width, not from `NSOutlineView`'s current frame width. On launch, the outline can hold a stale frame width even after the split view has restored the real sidebar size, which leaves trailing markers and project `+` buttons flush against the outer edge until a manual divider drag forces a resize.
- Avoid per-layout `noteHeightOfRows(...)` invalidation, which caused visible resize lag/flicker during divider drags.
- `ThreadListViewController.reloadData()` rebuilds the full outline tree during periodic thread/session refreshes, so it must capture and restore the `NSScrollView` clip-view origin around `outlineView.reloadData()` to preserve browsing position.
- When `reloadData()` re-selects the already-selected thread, suppress AppKit's internal `scrollRowToVisible(_:)` for that specific restore path and reapply the saved clip-view origin on the next main-queue turn as well. Immediate restore alone can still lose to deferred selection/layout scrolling.
- **Treat sidebar selection as controller state, not `NSOutlineView` state.** `reloadData()` and even metadata-only row refreshes can leave `NSOutlineView.selectedRow` temporarily invalid while AppKit rebuilds internal row objects. `ThreadListViewController` now owns `selectedThreadID`; real user/programmatic selections write through `recordSelectedThread(_:)`, while diff-panel and branch-mismatch refreshes resolve the current thread from that stored id. Do not clear the bottom-left `CHANGES` / `COMMITS` panel just because `outlineViewSelectionDidChange` reported `row = -1` during a refresh. Clear only when the controller-owned selection no longer resolves to a live thread.
- **Prefer direct in-place view updates over `reloadItem` for metadata-only changes.** `ThreadManagerDelegate.didUpdateThreads` fires whenever any thread property changes (including busy state, rate-limit, dirty flag every 5 s). Full `reloadData()` causes a render frame where the scroll position is wrong between `reloadData()` and the scroll restore callback — visible as a brief scroll-position blink. Use `ThreadListViewController.sidebarNeedsStructuralReload(for:)` to detect structural changes (threads added/removed/reordered/re-sectioned), and for metadata-only changes call `updateSidebarInPlace(with:)` instead. `updateSidebarInPlace` mutates `SidebarSection.threads` and `SidebarProject.children` in-place (both are class objects) then directly reconfigures existing visible row/cell views via `rowView(atRow:makeIfNecessary: false)` and `view(atColumn:row:makeIfNecessary: false)`. This preserves existing `AlwaysEmphasizedRowView` instances and their running CA animations (busy border rotation, shimmer). It does NOT call `reloadItem(reloadChildren: true)`, which would trigger `rowViewForItem` and create new row view instances, destroying animations.
- **Sticky header overlay is updated on every scroll and after every reload.** `ThreadListViewController` registers for `NSView.boundsDidChangeNotification` on the clip view. The `updateStickyHeaders()` method walks backwards from the first visible row to find the nearest project/section whose header row has scrolled off the top, then updates `StickyHeaderOverlayView`. `reloadData()` also calls `updateStickyHeaders()` (immediately and on the next main-queue turn alongside the scroll restore) so the overlay stays correct after structural changes.
- **`didUpdateThreads` calls are coalesced per run-loop cycle.** The session monitor can fire several `didUpdateThreads` delegate calls within a single tick (busy sync, rate-limit, completions — each awaits back to `MainActor.run` independently). Rather than refreshing the sidebar for each, `didUpdateThreads` stores the latest thread snapshot and schedules a single `applyThreadUpdate` via `DispatchQueue.main.async`. If another call arrives before the scheduled work fires, the snapshot is replaced and the previous work item is cancelled. `didCreateThread`, `didArchiveThread`, and `didDeleteThread` remain immediate (no coalescing) since they need synchronous UI effects like auto-selecting the new row.
- **Launch centering of the selected thread must be deferred until view + data readiness.** The launch-only centering pass should run once, only after `viewDidAppear` and at least one settled sidebar reload/layout pass. If row geometry is not ready yet (no rows/zero visible height), retry on the next main-queue turns for a bounded number of attempts. Do not center on every reload — launch-only.

## Gotchas

- Do not key sidebar row-height math off `hasUnreadAgentCompletion` or other selection-sensitive flags unless the visible layout is guaranteed to stay identical before and after selection.
- Dynamic row heights must be deterministic for a given thread state — the same thread data must always produce the same height. If height changes with selection, the outline view will visually jump.
- A split-view width fix alone is not sufficient. The sidebar can look like it is resizing when the real bug is row-height changes inside cells.
- Reloading the outline without restoring scroll position will look like a spontaneous scroll-position blink, especially because session-monitor updates fire every few seconds even when the user is not interacting with the sidebar. Use the structural-vs-metadata split (`sidebarNeedsStructuralReload`) to avoid full `reloadData()` for routine status updates.
- **Structural keys that determine when a full reload is needed**: `id`, `sectionId`, `displayOrder`, `isPinned`, `isSidebarHidden`, `isArchived`, and `lastAgentCompletionAt` **only when `autoReorderThreadsOnAgentCompletion` is enabled**. When that setting is off, completion date does not affect display order and must not be included in the key — otherwise every agent completion triggers a structural reload and the animate-open jitter described below. Everything else (busy sessions, rate-limit, dirty flag, PR info, task description, icons, branch state) is metadata-only and can be updated in-place.
- **Suppress animations in the restore loop**: `reloadData()` creates new `SidebarProject`/`SidebarSection` class instances on every call. NSOutlineView tracks expansion state by object identity, so after `outlineView.reloadData()` all new items start in the collapsed state. The restore loop then calls `expandItem()` on each one — and with the default animation duration this produces a visible slide-open animation for every section on every reload. Wrap the entire restore loop in `NSAnimationContext.beginGrouping()` / `NSAnimationContext.current.duration = 0` / `NSAnimationContext.endGrouping()` so expansions are instantaneous and the user never sees sections "pop" open.
- Re-selecting the current row during a background reload can still trigger AppKit auto-scroll even after an immediate scroll restore. Keep any `scrollRowToVisible` suppression narrowly scoped to internal selection-preservation paths so normal user-driven selection still reveals the chosen row.
- `SplitViewController`, notification-driven navigation, and any future CLI/IPC selection entry points must go through `ThreadListViewController.selectThread(byId:)` or `recordSelectedThread(_:)` so controller-owned selection state and the visible outline selection never drift apart.
- **Trailing icons can appear flush to the edge on first launch** — this is a transient initial-layout-pass timing issue (frames settle on first resize) and not a constraint logic bug. It does not recur after the first sidebar resize or window layout pass. Do not mistake it for a regression introduced by unrelated changes to views anchored below the outline view (e.g. `DiffPanelView`).
- **Do not constrain `DiffPanelView`'s internal `stackView` to `scrollView.contentView` anchors** — the clip-view frame is managed internally by `NSScrollView` and mixing auto layout constraints into that chain causes the outline view's trailing markers to momentarily snap to the wrong position and fail to track sidebar resize drag. Constrain to the `scrollView` frame anchors instead (leading/trailing), and use `.defaultLow` horizontal compression resistance on any variable-width labels inside document-view rows to prevent them from pushing the scroll view wider.
- **Do not leave the diff-panel tab bar at default horizontal resistance** — when the bottom-left panel reveals `COMMITS (n)`, AppKit can treat the tab controls as a larger fitting width and silently clamp divider dragging. Keep the tab buttons and tab stack on low horizontal hugging/compression resistance so the sidebar still honors the split view's intended minimum.
- **`shouldExpandItem`/`shouldCollapseItem` must only permit the programmatic restore loop**: NSOutlineView calls these for keyboard navigation, type-select auto-expansion, and — critically — for internal triggers with a nil `currentEvent` (e.g. during `reloadItem`, run-loop coalescing, or background AppKit callbacks). The previous code returned `true` for all non-keyboard events, meaning AppKit could expand/collapse items on its own initiative at any time. Both methods now unconditionally `return false` unless `allowsProgrammaticOutlineDisclosureChanges` is true. That flag is set exclusively by `reloadData()`'s restore loop, ensuring sections and projects can only change visual state via our persistence-backed path.
- **`shouldSelectItem` must bail out for non-mouse events before acting on sections**: `sectionHeaderHitArea` returns `.other` for any event that is not `leftMouseDown`/`leftMouseUp` (keyboard navigation, type-select, programmatic selection). The `.other` branch in `shouldSelectItem` calls `toggleSection`, which would inadvertently collapse/expand sections whenever arrow-key navigation crosses a section row. Guard the section handling block with `NSApp.currentEvent?.type == .leftMouseDown || .leftMouseUp` and return false immediately for any other event type. Without this guard, the keyboard fix in `shouldExpandItem`/`shouldCollapseItem` makes the bug visible: the toggle from `shouldSelectItem` now sticks (instead of being masked by the native expand that was previously allowed).
- **`outlineViewItemDidCollapse`/`outlineViewItemDidExpand` must be no-ops during `reloadData()`**: AppKit fires `NSOutlineViewItemDidCollapseNotification` (and thus `outlineViewItemDidCollapse`) for previously-expanded `SidebarProject` items when the outline view resets its expansion state during `reloadData()`. Without a guard, this writes those projects into `collapsedProjectIdsKey` in UserDefaults, causing the restore loop immediately after to see them as collapsed and collapse them — hiding all their sections. Guard both callbacks with `guard !isReloadingData else { return }`. Also read `collapsedIds` from UserDefaults **before** calling `outlineView.reloadData()` so the snapshot is taken from the correct pre-reload state.
- **`selectedThreadID` must survive section collapse**: When the currently selected thread is inside a section the user collapses, `reloadData()` cannot find that thread in the outline view's visible rows and would previously clear `selectedThreadID`. This caused the next `didUpdateThreads` tick (via `outlineView.selectedRow < 0`) to call `autoSelectFirst()`, jumping focus to the Main thread. Fix: only clear `selectedThreadID` when the thread no longer exists in `threadManager.threads` (truly deleted/archived), not merely when it's hidden inside a collapsed section. Likewise, guard the `autoSelectFirst()` call in `didUpdateThreads` with `selectedThreadFromState() == nil` so a hidden-but-valid thread is never evicted by a background tick.
- **Cancel pending section-name toggle on thread selection**: `scheduleSectionNameToggle` defers a `toggleSection` call by `NSEvent.doubleClickInterval` (~0.5 s) when the user single-clicks a section name. If the user then selects a thread (or submits a prompt that triggers sidebar activity) within that window, the deferred toggle still fires and unexpectedly collapses/expands the section. Cancel any pending toggle in `outlineViewSelectionDidChange` so a thread selection always wins over a stale single-click.
- **`updateSidebarInPlace` rowView block must mirror every property set in `rowViewForItem`**: When adding a new `AlwaysEmphasizedRowView` property that's set during initial row creation (`rowViewForItem`), it must also be set in the `updateSidebarInPlace` rowView block — otherwise the property is only applied on full structural reloads and never cleared/updated during metadata-only updates. `showsRateLimitHighlight` was missing from the in-place path, which caused the rate-limit red border to persist after `markSessionRateLimitSeen` cleared the unread state.
- **NSOutlineView can hand delegate methods stale value-type thread snapshots after metadata-only updates**: `MagentThread` is a struct. During `updateSidebarInPlace` we refresh visible rows without `reloadData()`, so AppKit may keep older boxed `MagentThread` items for row virtualization and pass them back later (for example when rows are recycled after scrolling). In thread-facing delegate paths (`rowViewForItem`, `viewFor`, `heightOfRowByItem`, and selection gating), resolve the thread from the latest `threadManager.threads` snapshot by `id` before deriving visual state.
- **`reloadItem(_:reloadChildren:true)` destroys row views and their CA animations**: `rowViewForItem` always creates a new `AlwaysEmphasizedRowView`, so `reloadItem` discards running busy-border and shimmer animations. It also clears NSOutlineView's visual row selection. For metadata-only updates, `updateSidebarInPlace` now avoids `reloadItem` entirely — it reconfigures existing row/cell views directly. If `reloadItem` must be used in the future, animation continuity requires storing phase info in the class-level `sharedBorderAnimationStartTimes` dictionary (keyed by `busyBorderPhaseKey`).
